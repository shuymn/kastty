# 0016: ターミナルバッファをオーバーレイ PTY で `$EDITOR` に開く

## Status

Accepted

## Context

ブラウザ上のターミナルバッファ／スクロールバックを、ユーザーの手元のエディタで開いて編集・保存したいという要求がある。実現方法として以下の選択肢があった：

1. 外部 GUI エディタを別プロセスとして起動する
2. メイン PTY にエディタ起動コマンドを注入する
3. エディタ専用の第 2 PTY を起動し、ブラウザ内オーバーレイとして描画する

1 は kastty が「ブラウザ内で完結する」という前提と矛盾し、リモート越し利用で破綻する。2 はメイン PTY 上で任意のフォアグラウンドプロセスが動作しうるため、コマンド注入は安全でなく挙動も予測不能になる（[0005](0005-blocking-command-pty-lifecycle.md) のライフサイクル前提も崩れる）。

`nvim` / `vim` のような TUI エディタを正しく動かすには、エディタ専用の PTY と独立した入出力チャネルが必要になる。

## Decision

エディタ機能を**独立したエディタ PTY セッション**として実装する。

- メインターミナルとは別に、専用 WebSocket ルート `/editor-ws` とエディタ専用の `PtyAdapter` を持つ `EditorSessionManager` を設ける。同時に開けるオーバーレイは 1 つだけとする。
- `/editor-ws` への接続だけではエディタを起動しない。クライアントが `{ t: "editor-open", content }` 制御メッセージを送って初めて、サーバが一時ファイルを作成し（`content` = 抽出したバッファテキスト）、エディタ PTY を**遅延起動**する。これにより「起動前にバッファ内容を書き込む」順序を保証する。プロトコルのフレーム規約は [0004](0004-websocket-protocol-design.md) を踏襲する。
- エディタコマンドは `$VISUAL` を優先し、無ければ `$EDITOR` を使う。両方が空なら起動せずエラーを返す。起動は素朴な空白分割ではなく、制御されたシェル起動 `/bin/sh -c '<EDITOR値> "$@"' kastty-editor <一時ファイル>` を用いる。エディタ値はスクリプトへ展開してユーザーの引数・クォートを保持し、一時ファイルパスは positional パラメータとして渡してインジェクションを防ぐ（`git` の `GIT_EDITOR` 起動と同じ定石）。
- 入力分離は二重ガードとする。オーバーレイ端末を `focus()` し、加えてメイン端末の `onData` に「オーバーレイがアクティブなら送信しない」ガードを置く（[0006](0006-readonly-double-guard.md) と同じ二重防御の発想）。
- 一時ファイルはサーバが所有し、エディタ PTY 終了時に削除する。削除失敗はログのみでメインセッションに影響させない。
- `editor-open` の `content` はバッファ全体（スクロールバック＋画面）を運ぶため正当な入力でも大きくなりうる。サイズ上限はプロトコルスキーマではなく WebSocket トランスポート層（`maxPayloadLength`）で一本化する。値は固定せず、設定済みスクロールバックのバイト予算（replay buffer と同じ `replayBufferBytes`。デフォルト 50000 行で約 98 MiB）から導出し、大きな履歴設定でも正当なスナップショットが無言ドロップされないようにする。未指定時のフォールバックは定数 `MAX_WS_PAYLOAD_BYTES`（16 MiB）。この上限は `createServer()` が返す `websocket` ハンドラに載せ、すべての `Bun.serve()` 呼び出し側で同じ値が適用されるようにする。スキーマで解釈不能な制御メッセージ（型不一致など）は `error` 通知後に接続を閉じ、オーバーレイを `connecting` のまま放置しない。
- バッファ抽出は normal バッファ（スクロールバック＋画面）を対象とし、ソフトラップ行を論理行へ再結合する。読み取り専用 API のみを使い、選択・スクロール位置を変更しない。ANSI 色／スタイルは保持しない。
- セキュリティ境界はメイン WebSocket と同一とする。`/editor-ws` も localhost 限定（[0002](0002-localhost-only-security-model.md)）とトークン検証（[0007](0007-url-query-token.md)）を必須とし、未認証の PTY 起動・接続を許可しない。
- 起動ショートカットは `Ctrl+Shift+E` に固定する。ブラウザのコピー／ペースト（`Ctrl+Shift+C/V`）と衝突せず、主要ブラウザで未割り当てのため。レイアウト非依存にするため物理キー（`event.code === "KeyE"`）で判定する。

## Consequences

### Positive

- メイン PTY を一切汚染せず、TUI エディタも含めて安全にバッファ編集ができる
- エディタチャネルがメインと完全分離され、関心の分離が保たれる（ルートのパスでチャネルを判別）
- 制御されたシェル起動により、ユーザー指定の引数を保ちつつパスインジェクションを防げる
- 同時 1 セッション制約と遅延起動により、レースや多重 PTY 起動を構造的に防げる

### Negative

- メインとエディタのチャネルが同一のプロトコル型を共有するため、メッセージの帰属は WebSocket のパスでしか区別できない
- 巨大スクロールバックでは `editor-open` ペイロードが大きくなりうる（現状は上限を設けていない）
- ソフトラップ判定は ghostty-web の `isWrapped` に依存し、これが正しく立たない場合は折り返し行が分割されたまま抽出される

### Neutral

- ショートカットは設定可能にせず固定とする。エディタ選択は `$VISUAL` / `$EDITOR` で既に設定可能であり、ショートカットの設定化は複雑さに見合わないと判断した
- 複数オーバーレイの同時表示、外部 GUI エディタ起動、タブ／ペイン管理は本決定のスコープ外とする
